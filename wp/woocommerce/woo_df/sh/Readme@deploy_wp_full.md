## abstract

部署指定网站包的设计

* 先从部署单个站点的实现入手
* 对于需要指定多个站点的方式,可以考虑用循环实现,无论是内部循环还是外部调用此脚本循环处理

单个站点的部署方案:
- 当前的部署思路是:根据指定的压缩包的所在目录(pack_root),扫描人员目录(提取为人员变量username),扫描人员目录下的压缩包组(archive),部署(deploy)网站文件(计算并提取包文件名中的网站名(通常是域名domain_name),解压压缩包组,并移动必要文件到指定目录(网站根目录),导入数据库,以及额外的修改收尾操作)
> deploy_site函数负责解压,其接收参数有:user_name,domain_name
- 分析:
  - 如果被指定的包路径符合上面的路径规律(包名domain_name和包文件父目录名是所属人员名username),那么其实用户只需要提供完整包路径即可,脚本中自行提取这些要素,就能将网站部署到正确位置;
  - 如果要更灵活,也就网站包路径不符合上述规律也能处理,那么需要用户通过参数指定其他必要参数(例如所属人员,甚至网站的域名.)
  - 如果要同时支持上述这些情况,由于要执行的操作目的最终是一样的,可以考虑将两种情况做一个统一处理,再根据这个统一的状态设计统一的部署操作;
  - 向更灵活,准确的表示方式看齐,那么对于符合规律的路径指定的方式虽然紧凑但是显然是要更严格的;
  - 我们考虑将提供完整路径情况下,解析其中的所属人员,域名等信息,并生成一个统一表示,再根据这个统一表示进行部署操作;
为了代码逻辑更加统一,减少代码嵌套深度,即便将内部循环搬出来,在外部遍历任务列表
